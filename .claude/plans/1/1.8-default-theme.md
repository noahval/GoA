# Default Theme

**Goal**: Create unified UI theme resource (default_theme.tres) with orange/blue/red color scheme, clear button states, and consistent styling across all scenes

**Success Criteria**:
- default_theme.tres created with complete style definitions
- Orange/blue/red color scheme implemented
- Six button types defined: Cancel, Affirmative, ForwardNav, BackNav, Danger, Muted
- All button states defined: Normal, Hover, Pressed, Disabled
- Font sizes reference ResponsiveLayout (Phase 1.10)
- Panel opacity: 30% for translucent overlay aesthetic
- Theme applied globally via project settings

**Prerequisites**:
- Phase 1.1 (Project Setup) - Project and resolution configured
- Phase 1.10 (Responsive Layout) - Font sizing and button dimensions defined

---

## Overview

Establishes the visual foundation for GoA's UI. The default theme provides consistent colors, typography, and interaction states across all scenes. Uses an orange/blue color palette that reinforces action hierarchy: orange for standard/negative actions, blue for affirmative/navigation actions.

### Key Design Principles

- **Orange/blue/red color scheme**: Orange base, blue accents for affirmative/nav, red for danger
- **Visual hierarchy through color**: Orange = standard, Blue = affirmative/navigation, Red = danger/destructive
- **Border thickness communicates importance**: 2px forward nav > 1px back nav
- **Translucent overlays**: 30% opacity maintains layered aesthetic
- **Unified font size**: Defined by ResponsiveLayout (Phase 1.10), 25px base, scales automatically
- **Obviously disabled**: Grey buttons with low opacity, clearly unclickable
- **Orange theme variations**: Stick with orange for now, can create regional theme variations later as needed

### Design Philosophy

**Why Orange/Gold/Blue?**
- **Orange** = Warm, attention-grabbing, cancel/decline/negative actions
- **Gold/Amber** = Valuable, rewarding, affirmative/confirm actions (fits furnace theme - molten metal, reward)
- **Blue** = Cool, trustworthy, navigation actions
- Creates natural visual hierarchy without thinking

**Why Gold for Affirmative?**
- Gold = reward, success, valuable outcome (semantically fits "confirm/accept")
- Distinct from orange (more yellow, clearly different hue)
- Distinct from blue navigation buttons (completely different color family)
- Thematically fits industrial/furnace aesthetic (brass, gold coins, molten metal)

**Why Different Blues?**
- **Light blue (forward nav)** = Progress, advancement, moving deeper
- **Navy blue (back nav)** = Retreat, return, going back
- Directional color temperature (light = forward, dark = back)

**Why Border Thickness Matters?**
- **2px border (forward nav)** = Emphasizes primary action
- **1px border (back nav)** = De-emphasizes secondary action
- Thickness creates hierarchy at a glance

## Button Types

### 1. CancelButton (Orange)

**Usage**: Cancel, decline, neutral, or negative actions. This is the default Button style.

**States**:
```gdscript
# Normal
background: Color(0.361, 0.180, 0, 0.2)       # Dark orange, 20% opacity
border: 1px, Color(0.6, 0.38, 0.15, 0.45)
corner_radius: 3px
text_color: Color(1, 1, 1, 1)                 # White

# Hover
background: Color(0.42, 0.25, 0.05, 0.2)      # Lighter orange, 20% opacity
border: 1px, Color(0.7, 0.45, 0.2, 0.5)       # Brighter border
text_color: Color(1, 1, 0.8, 1)               # Light yellow

# Pressed
background: Color(0.30, 0.15, 0, 0.25)        # Darker orange, 25% opacity
border: 1px, Color(0.5, 0.3, 0.1, 0.5)
text_color: Color(0.9, 0.9, 0.9, 1)           # Light grey

# Disabled
background: Color(0.3, 0.3, 0.3, 0.15)        # Grey, 15% opacity
border: None or 1px subtle grey
text_color: Color(0.5, 0.5, 0.5, 0.6)         # Dimmed grey
```

**Examples**: Cancel, Back, Close, Delete, No

**Usage**: Low-prominence utility buttons that shouldn't draw attention (settings, options, secondary actions)

**States**:
```gdscript

border: 1px, Color(0.5, 0.5, 0.5, 0.5)           # Slightly brighter border
text_color: Color(1, 1, 1, 1)                    # White on hover

# Pressed
background: Color(0.2, 0.2, 0.2, 0.4)            # Darker grey, 40% opacity
border: 1px, Color(0.35, 0.35, 0.35, 0.4)        # Darker border
text_color: Color(0.75, 0.75, 0.75, 1)           # Dimmed text

# Disabled
background: Color(0.2, 0.2, 0.2, 0.15)           # Very low opacity
```

**Visual Features**:
- **1px border** (subtle, not attention-grabbing)
- **Neutral grey** (no color temperature - intentionally plain)
- **Lower contrast** than other buttons (recedes visually)
- **Light grey text** (not pure white, further reduces prominence)

**Examples**: Settings, Options, Preferences, Help, Info, About

**Usage Guidelines**:
- Use for utility/meta actions that aren't part of core gameplay
- Settings button in every scene should use this style
- Keeps focus on gameplay buttons (orange/blue) while still providing access

### ProgressBar

**Template Structure**:

All progress bars follow this uniform template:

```gdscript
# Background (empty part of bar)
background: Color(0.2, 0.2, 0.2, 0)           # Transparent dark grey
border: 1px, Color(0.3, 0.3, 0.3, 0.5)        # Optional subtle border
corner_radius: 2px
height: 20px (scaled with resolution)

# Fill (filled part of bar)
fill_color: [SEE VARS BELOW]                  # Color varies by bar type
corner_radius: 2px
opacity: 30%

# Label (centered text inside bar)
label_font_size: 14px                          # Smaller than base 25px for compact bars
label_alignment: center (horizontal + vertical)
label_anchors: PRESET_FULL_RECT               # Fill entire bar area
```

**IMPORTANT: Font override interaction with ResponsiveLayout**

Labels with custom `font_size` overrides are preserved by `ResponsiveLayout.apply_to_scene()`. The layout system checks `has_theme_font_size_override()` before applying the default scaled font size. This means:
- Set font_size override in .tscn: `theme_override_font_sizes/font_size = 14`
- Set font_size override in code: `label.add_theme_font_size_override("font_size", 14)`
- ResponsiveLayout will skip these nodes and keep your custom size

**How to Use**:
1. Create ProgressBar node in scene
2. Apply default theme (handles background/structure)
3. Reference a named bar from vars below
4. Set fill color via theme override or StyleBoxFlat

- `MIN_BUTTON_WIDTH` (120px)

### Theme Implementation

The theme does NOT specify font sizes directly. Instead:
- Font sizes are applied automatically by `ResponsiveLayout.apply_to_scene()`
- All Labels, Buttons, and RichTextLabels get `FONT_NORMAL` (25px scaled) by default
- Manual overrides only for special cases (titles, debug text, etc.)

**Key Insight**: ResponsiveLayout handles ALL font scaling. The theme only handles colors, borders, padding, and visual styling.

## Button Animation & Interactivity

**NOTE**: This section will be fleshed out in a future implementation phase.

### Animation Requirements

Button state transitions (Normal → Hover → Pressed) currently have no animation specified. Future implementation should address:

1. **Transition Duration**: How long does color/scale change take? (Suggested: 0.1-0.2s)
2. **Easing Function**: Linear? Ease-out? (Suggested: ease-out for natural feel)
3. **Properties to Animate**:
   - Background color fade
   - Text color fade
   - Border color fade
   - Shadow size/offset (for ForwardNav buttons)
   - Scale (optional micro-scale on press: 0.98x)

4. **Implementation Method**:
   - Tween-based animation in code?
   - AnimationPlayer nodes?
   - Shader-based transitions?
   - Theme doesn't natively animate - requires scripting

### Shadow Performance

**NOTE**: Shadow performance optimization will be addressed in the animation section.

ForwardNav buttons use StyleBoxFlat shadows that animate on hover/press. Considerations:

1. **Performance Cost**: Multiple shadows can impact frame rate on low-end systems
2. **Testing Needed**: Profile with 20+ buttons on screen simultaneously
3. **Fallback Options**:
   - Make shadows optional in settings
   - Disable shadows below certain FPS threshold
   - Use alternative visual effect (border glow) instead

4. **Implementation Notes**:
   - Document shadow performance in testing phase
   - Consider togglable "reduced motion" accessibility option

## Implementation Tasks

### 1. Create Theme Resource

**Steps**:
1. In Godot Editor: FileSystem → Right-click root
2. Create → Resource → Theme
3. Save as: `res://default_theme.tres`

### 2. Define Base Button Styles

**In default_theme.tres** (via Inspector or text editor):

Create StyleBoxFlat resources for each button type and state:
- StyleBoxFlat_button_normal (orange)
- StyleBoxFlat_button_hover (orange)
- StyleBoxFlat_button_pressed (orange)
- StyleBoxFlat_button_disabled (grey)

Apply to Button type in theme editor.

### 3. Create Theme Variations

**For specialized buttons**, create type variations:

```gdscript
# In default_theme.tres
[resource]
AffirmativeButton/base_type = &"Button"
AffirmativeButton/styles/normal = SubResource("StyleBoxFlat_affirmative_button_normal")
AffirmativeButton/styles/hover = SubResource("StyleBoxFlat_affirmative_button_hover")
AffirmativeButton/styles/pressed = SubResource("StyleBoxFlat_affirmative_button_pressed")
AffirmativeButton/styles/disabled = SubResource("StyleBoxFlat_button_disabled")

ForwardNavButton/base_type = &"Button"
ForwardNavButton/styles/normal = SubResource("StyleBoxFlat_forward_nav_button_normal")
# ... etc

BackNavButton/base_type = &"Button"
BackNavButton/styles/normal = SubResource("StyleBoxFlat_back_nav_button_normal")
# ... etc

DangerButton/base_type = &"Button"
DangerButton/styles/normal = SubResource("StyleBoxFlat_danger_button_normal")

DangerButton/styles/pressed = SubResource("StyleBoxFlat_danger_button_pressed")

```


**Back Nav Button**:

## Design Rationale

### Why This Color Scheme?

**Orange base**:
- Warm, industrial, matches furnace/coal theme
- Attention-grabbing without being alarming
- Good contrast with blue accents

**Blue accents**:
- Cool, trustworthy, clear distinction from orange
- Light blue = forward (optimistic, progress)
- Navy blue = back (retreat, safety)

**30% opacity**:
- Translucent overlays maintain visual layering
- Background visible through UI (immersive)
- High enough for readability, low enough for depth

### Border Strategy

**Navigation buttons get prominent borders**:
- Forward: 2px (primary action)
- Back: 1px (secondary action)
- Creates instant visual hierarchy

**Standard/affirmative: subtle or no border**:
- Focus is on color, not outline
- Cleaner appearance

### Shadow Usage

**Forward nav only**:
- Adds depth, makes button "pop"
- Reinforces it's the primary action
- Animates on hover/press (tactile feedback)

**Others: no shadow**:
- Keeps design clean
- Prevents visual clutter

## Notes & Decisions

**Decision 1**: Orange/Gold/Blue scheme
- **Rationale**: Matches industrial/furnace aesthetic
- **Benefit**: Clear visual language (orange = cancel/negative, gold = affirmative/positive, blue = navigation)
- **Gold choice**: Distinct from both orange and blue, thematically fits (molten metal, reward, valuable)

**Decision 2**: Four button types instead of two
- **Rationale**: Navigation needs visual distinction from dialog actions
- **Benefit**: Players instantly know forward vs back, confirm vs cancel

**Decision 3**: Border thickness communicates hierarchy
- **Rationale**: Subtle visual cue that doesn't require color perception
- **Accessibility**: Works for colorblind players

**Decision 4**: Obviously disabled state (grey, low opacity)
- **Rationale**: Disabled buttons must be unmistakably inactive
- **Implementation**: 15% opacity grey, dimmed text, no hover effect

**Decision 5**: Unified 25px base font
- **Rationale**: Simplicity, consistency, scales with resolution
- **Benefit**: One size to maintain, easy to adjust globally

**Decision 6**: 30% panel opacity
- **Rationale**: Sweet spot between readability and translucency
- **Tested**: Lower = too transparent, higher = loses layered aesthetic
